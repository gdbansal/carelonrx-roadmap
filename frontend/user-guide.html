1. Introduction to Product 360

Product 360 is an integrated product suite that combines multiple tools for comprehensive product management. The platform provides a unified experience with a persistent side panel for seamless navigation between modules.

Product 360 Modules

Requirements Intake (Available Now)

Structured intake form for capturing product initiatives with WSJF scoring, priority, budget, and dependency tracking.

Roadmap Module (Available Now)

Product initiative management system for planning, tracking, and visualizing initiatives across quarters and years.

Capacity Planning Module (Available Now)

PI-based sprint capacity planning tool for teams to manage working days, availability, and suggested story points per sprint with optional JIRA sprint integration.

Story Mapping (Available Now)

AI-powered user story mapping � fetches JIRA epics and Confluence pages, then breaks them down into INVEST-compliant stories with JIRA ticket creation support.

LOE Estimation Module (Available Now)

Level of Effort estimation per initiative � break down effort by system with Dev, QA, and Support story points, toggle between SP and Dollar modes, and export to CSV.

Story Estimations Module (Available Now)

Collaborative Planning Poker tool for agile teams to estimate user stories using Fibonacci sequence with real-time collaboration.

Key Features

  • Unified Navigation: Side panel for seamless module switching
  • Initiative Management: Complete intake and tracking system
  • WSJF Prioritization: Weighted Shortest Job First methodology
  • Visual Timeline: Interactive roadmap across quarters
  • Requirements Intake: Structured capture of product initiatives
  • Story Estimations: Planning Poker with real-time collaboration
  • Story Mapping: AI-powered INVEST story breakdown from JIRA epics and Confluence pages
  • LOE Estimation: Per-initiative effort estimation by system in story points or dollars with CSV export
  • Analytics Dashboard: Real-time statistics and charts
  • Role-Based Access: Dynamic role permissions with module-level visibility control
  • Budget Tracking: Approval status management
  • Dependency Management: Track system dependencies
  • Audit Logging: Comprehensive tracking for estimations
  • Capacity Planning: PI-based team capacity matrix with JIRA sprint integration
  • Team Availability: Auto-computed availability and suggested story points per sprint

7. Creating Initiatives

Step-by-Step Guide

  1. Click "New Initiative" button
  2. Fill in the Basic Information:
    • Initiative Name (required)
    • Description (required)
    • Program (required)
    • Year (required): 2025-2030
    • Quarter (required): Q1-Q4
    • Line of Business (required): Select from admin-configured LOBs
    • Start Date (optional)
    • Delivery Date (optional)
    • Business Commitment Date (optional)
  3. Set Priority (required): Critical, High, Medium, Low, or On Hold
    • If On Hold is selected, a Hold Reason field appears (required, max 500 chars)
  4. Enter WSJF Components (Fibonacci numbers: 1, 2, 3, 5, 8, 13, 21):
    • User-Business Value
    • Time Criticality
    • RR/OE (Risk Reduction/Opportunity Enablement)
    • Job Size
    • WSJF Score: Automatically calculated
  5. Add Business Owner (required)
  6. Enter Business Value and Risks
  7. Add Dependent Systems (optional)
  8. Set Budget Status (Approved, Pending, or Rejected)
  9. Click "Submit Initiative"

WSJF Calculation Formula:

WSJF = (User-Business Value + Time Criticality + RR/OE) / Job Size

Example:

  • User-Business Value: 13
  • Time Criticality: 8
  • RR/OE: 5
  • Job Size: 3
  • WSJF = (13 + 8 + 5) / 3 = 8.67

7. Story Mapping Module

The Story Mapping module uses AI to break down JIRA epics and Confluence pages into INVEST-compliant user stories, then lets you create those stories as JIRA tickets.

How It Works (5 Steps)

Step 1 — Input

Provide a JIRA Epic URL and optionally a Confluence page URL as source material.

Step 2 — Fetch

The system fetches content from JIRA and Confluence via the backend proxy (requires corporate network or internal server).

Step 3 — Analyze

AI analyzes the fetched content and produces an INVEST-compliant story breakdown with acceptance criteria.

Step 4 — Review

Review the generated stories, approve or remove individual items before creating tickets.

Step 5 — Create Tickets

Approved stories are pushed as JIRA sub-tasks or stories under the parent epic.

Note:

JIRA and Confluence integration requires the backend to be running on the corporate network (internal server) so it can reach jira.carelonrx.com and confluence.carelonrx.com.

9. Story Estimations Module

The Story Estimations module provides agile story point estimation using the Fibonacci sequence for sprint planning and team collaboration.

? Now Available!

Story Estimations is fully integrated with MongoDB backend for real-time collaboration and data persistence.

7.1 Creating a Session

Start a new estimation session for your team:

  1. Click "Story Estimations" in the Product 360 sidebar
  2. On the login screen, enter:
    • Name: Your name
    • Role: Select from Dev, QA, PO (Product Owner), or SM (Scrum Master)
  3. Click "Login"
  4. Click "Create New Session"
  5. Fill in session details:
    • Team Name: Your team name (e.g., "Team Alpha")
    • Sprint Value: Sprint identifier (e.g., "Sprint 23")
  6. Click "Create"

Session ID:

After creating a session, you'll receive a unique Session ID. Share this ID with team members so they can join your session.

7.2 Joining a Session

Join an existing estimation session:

  1. Get the Session ID from the session creator
  2. Enter your Name and select your Role
  3. Click "Login"
  4. Enter the Session ID in the "Join Session" field
  5. Click "Join"

?? Product Owner Restriction:

Only one Product Owner (PO) is allowed per session. If a PO already exists, you must choose a different role.

7.3 Adding Stories

Add user stories to estimate:

  1. In the "Add Story" section, enter:
    • Story Number: JIRA ticket number or story ID
    • Story Summary: Brief description of the story
  2. Optionally, add a JIRA Link URL for the story number
  3. Click "Add Story"

Story Restrictions:

  • Dev Only: Only developers can estimate
  • QA Only: Only QA can estimate
  • All Roles: Everyone can estimate (default)

7.4 Estimating Stories

Provide story point estimates using Fibonacci sequence:

Fibonacci Points Available:

0, 0.5, 1, 2, 3, 5, 8, 13

  1. Find the story you want to estimate
  2. Click on the appropriate Fibonacci number button
  3. If prompted (for Dev/QA roles), select a reason for your estimate
  4. Your estimate is saved automatically

Estimation Reasons:

Developers and QA can provide reasons for their estimates:

  • Dev: "Dev analysis is more", "Development is more", etc.
  • QA: "QA Analysis will take more time", "Test cases are more", etc.

Viewing Estimates:

  • Your own estimates are always visible to you
  • Other team members' estimates are hidden until revealed by PO
  • Average estimates shown after reveal

7.5 Revealing Estimates

Product Owner Only: Reveal all team estimates for a story.

  1. Wait for all team members to submit their estimates
  2. Click "Reveal Estimates" button for the story
  3. All estimates become visible to everyone
  4. View breakdown by role (Dev average, QA average)
  5. See overall average estimate
  6. Review estimation reasons provided by team members

After Revealing:

  • Team can discuss differences in estimates
  • Re-estimate if needed
  • Mark story as complete when consensus reached
  • Export summary for documentation

Export Options:

  • Download Summary: Export all estimates to Excel
  • Story Analysis: Includes averages by role
  • Deviation Analysis: User estimation patterns

11. Capacity Planning Module

The Capacity Planning module enables teams to plan sprint-level resource availability per Program Increment (PI), with optional JIRA sprint integration to auto-populate sprint columns.

Access:

Navigate via the Product 360 sidebar ? Capacity Planning

11.1 Selecting PI, Team & Project

Before loading the capacity matrix, select the following filters:

Required fields (marked with *):

  • Program Increment (PI):* Auto-defaults to the current quarter (e.g., 26.3 for Q3 2026). Format is YY.Q. Covers 2 past and 4 future PIs.
  • Line of Business (LOB):* Select the LOB from the dropdown.
  • Project:* Select the project — options are filtered based on the selected LOB.
  • Team:* Select the team — options are filtered based on the selected project.

Click

Click "Load Team Members" to populate the capacity matrix.

11.2 Capacity Matrix

The matrix displays all team members with editable capacity inputs per sprint:

Table structure:

  • PI Banner: Orange header row showing the selected PI (e.g., PI 26.3) spanning all columns.
  • Title: Shows Capacity Matrix � [Team Name] above the table.
  • Columns: Resource Name | Role | Capacity % | Sprint 1 � Sprint N (dynamic)
  • Capacity %: Editable field (0-100) indicating the percentage of time this member is allocated to this team/project. Defaults to 100%. Saved to MongoDB with the capacity plan.
  • Capacity inputs: Enter working days (0�30) per member per sprint.
  • Average Working Days / Sprint: Auto-calculated average across all members for each sprint.
  • Resources available / Sprint: Count of team members excluding Dev Lead role � constant per sprint, auto-computed.

Sprint Columns:

Sprint columns default to Sprint 1 – Sprint 6 for the selected PI. The number of sprints is determined automatically when the team members are loaded.

11.3 Avg Points per Sprint

Below the capacity matrix, a summary table shows the average story points a person can complete per sprint when available 100%:

  • One editable input per sprint column � default value is 12.
  • Each value can be adjusted independently per sprint.
  • Changing a value instantly recalculates Suggested Points per sprint in the Team Availability table below.

11.4 Team Availability & Suggested Points

The Team Availability section provides per-sprint capacity analysis with the following columns:

Column Description Editable?
Working days (Team Avg) Pre-filled from capacity matrix average; manually adjustable. Yes
Admin Work % Percentage of time spent on admin tasks. Default: Sprint 1 = 5%, others = 10%. Yes
Contingency % Buffer percentage for unplanned work. Default: 5%. Yes
Availability (days/sprint) Auto-computed: Working days � (1 - Admin%) � (1 - Contingency%) No (computed)
Suggested Points per sprint Auto-computed: Availability � Resources (excl. Dev Lead) � (Avg Points � 10) No (computed)

Total Row:

The footer shows the Total Suggested Points � the sum of all sprints' suggested points for the PI. This is the team's estimated PI capacity.

Saving Capacity Plans:

Click "Save Capacity Plan" to persist the capacity data for the selected PI, team, and project. Saved plans store per-sprint capacity values and Capacity % for each team member in MongoDB.

Export to Excel:

After saving, an "Export to Excel" button appears. Clicking it downloads an .xlsx file containing:

  • Capacity Matrix — Resource Name, Role, Capacity %, per-sprint capacities, and average
  • Avg Points per Sprint — configured average story points per sprint
  • Team Availability — Working days, Admin %, Contingency %, Availability, and Suggested Points per sprint with a PI total

  • Columns = Modules: Roadmap, Story Estimations, Capacity Planning

12. Session Management

The application includes automatic session timeout for security.

Session Timeout

Duration: 30 minutes of inactivity

How It Works:

  1. Session timer starts when you login
  2. Timer resets with any user activity:
    • Mouse movements
    • Keyboard input
    • Clicks
    • Scrolling
  3. After 25 minutes of inactivity, a warning appears
  4. After 30 minutes, automatic logout

Session Warning

5-Minute Warning:

  • Appears at 25 minutes of inactivity
  • Modal dialog with two options:
    • "Stay Logged In": Extends session for another 30 minutes
    • "Logout Now": Immediately logs you out

Best Practices

  • Save your work frequently
  • Don't leave the application unattended
  • Click "Stay Logged In" if you see the warning
  • Log out manually when done

Initiative Data

  • Who: User who made the change
  • When: Timestamp of the change
  • What: Fields that were modified

Data Security

  • JWT Tokens: Secure authentication tokens
  • Session Management: Automatic timeout after 30 minutes
  • Role-Based Access: Permissions enforced at API level

14. Troubleshooting

Cannot Login

  • Check username and password
  • Ensure account was created successfully
  • Contact admin if account is locked

Email Validation Error

  • Use only @elevancehealth.com or @carelon.com emails
  • Check for typos in email address

Cannot Edit Initiative

  • You can only edit initiatives you created
  • Admins can edit all initiatives
  • Check if you're logged in with correct account

WSJF Not Calculating

  • All four WSJF fields are optional; score calculates only when all four are provided
  • Use only Fibonacci numbers (1, 2, 3, 5, 8, 13, 21)
  • Job Size cannot be 0 if provided
